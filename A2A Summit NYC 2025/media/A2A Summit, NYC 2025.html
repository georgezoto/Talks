The 4 Pillars of A2A

1. Universal Translation

JSON-RPC 2.0 payloads for vendor-agnostic messaging.

2. Capability Discovery

Agents broadcast skills via agent.json (The Agent Card).

3. Task Lifecycle

Managing state and "Artifact" hand-offs between agents.

4. Decentralized Trust

Standardized OIDC auth for secure collaboration.

Workflow in Action: Research to Deck

How A2A solves the hand-off between specialized agents.

AGENT 1
Researcher
OUTPUT: Report.pdf
➔
AGENT 2
Validator
OUTPUT: Verified_Data.json
➔
AGENT 3
Presenter
OUTPUT: Final_Deck.pptx

The "Hand-off" Logic:

  • Task Lifecycle: Agent 2 waits for Agent 1 to reach "status": "completed".
  • Artifacts: Agent 2 receives the artifact_id from Agent 1 as its input.
  • Autonomous Hire: The Orchestrator uses Agent Cards to find the best "Validator" for the specific research topic.

Engineering: The Task Request

Example of Agent 3 (Presenter) requesting data from Agent 2 (Validator).

{
  "jsonrpc": "2.0",
  "method": "createTask",
  "params": {
    "capability": "generate-presentation",
    "inputs": {
      "source_artifact": "artifact-882-verified-data",
      "template": "corporate-dark"
    },
    "callback_url": "https://orchestrator.internal/hooks/a2a"
  }
}

A2A vs. MCP

Choosing the right protocol for the right job.

Context MCP (The Tool Belt) A2A (The Network)
Connection Model ↔ Local Data/Files Agent ↔ Remote Agent
Integration Vertical (Internal) Horizontal (Cross-Org)
Example Reading your local GitHub Hiring a Legal Compliance bot

Strategic Roadmap for Product

  • Phase 1: Componentization. Break monolith features into "Agentic Services."
  • Phase 2: Capability Discovery. Publish agent.json for every internal micro-service.
  • Phase 3: Ecosystem Expansion. Allow 3rd-party A2A agents to trigger our workflows securely.

The Goal: A "Plug-and-Play" product architecture where our Researcher can work with ANY A2A-compliant Validator on the market.
